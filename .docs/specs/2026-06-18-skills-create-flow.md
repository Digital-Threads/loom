# SDD — Полировка Skills: флоу создания скилла + AI-generate

- **Дата:** 2026-06-18
- **Область:** только `loom-host/web` (фронтенд). Бэкенд и `src/core` **не трогаем**.
- **Файл изменения:** `web/src/components/Skills.tsx` (компонент `CreateSkill`).
- **Класс:** feature (полировка — доведение заглушки до рабочего состояния).

## 1. Контекст и проблема

Экран «Skills» (`web/src/components/Skills.tsx`) — библиотека Claude-скиллов из `~/.claude/skills`:
слева список с поиском, справа просмотр/правка `SKILL.md`. Создание скилла сделано «вчерне»
(заглушка `CreateSkill`): пользователь пишет описание, нажимает **Create**, фронт зовёт
`client.skillGenerate(desc, profile)` и **сразу** открывает результат по имени. Минусы текущей заглушки:

1. **Нет превью** — сгенерированный `SKILL.md` нигде не показывается перед открытием; возвращаемый
   бэкендом `content` отбрасывается (`web/src/api.ts:312` возвращает `{ name, content }`, используется только `name`).
2. **Нет валидации** — если агент вернул кривой скилл, пользователь это видит только постфактум; ошибка
   уходит в `toast`, контекст модалки теряется.
3. **Состояния busy/error поверхностные** — есть только надпись «Generating…» и тост на ошибку; во время
   генерации поля ввода не блокируются, ошибка не остаётся в модалке для повторной попытки.

### Что уже есть (факты из кода)

- API (`web/src/api.ts`):
  - `skills()` — список `SkillMeta[]`;
  - `skillGet(name) → { name, content }`;
  - `skillSave(name, content)` (PUT) — сохранить/создать;
  - `skillGenerate(description, profile?) → { name, content }` — AI-генерация.
- Бэкенд `POST /api/skills/generate` (`src/web/api.ts:1372`) **уже реализован** и **сразу записывает файл**
  (`generateSkill` в `src/core/skills/skills.ts`): прогоняет промпт через агента, парсит `name` из
  frontmatter, проверяет слаг, пишет `<name>/SKILL.md`, возвращает `{ name, content }`. На пустое описание
  отдаёт 400, на невалидный вывод агента — 422.
- Правило имени скилла (бэкенд `safeName`): `^[A-Za-z0-9._-]+$`, не начинается с `-`, без `..`
  (в промпте просят короткий **kebab-case** слаг).
- В `Skills.tsx` уже есть `splitFrontmatter(md)` → `{ description, invocable, body }` и компонент
  `Markdown` — ими рендерится просмотр скилла справа. Их же переиспользуем для превью.
- UI-кирпичи: классы `overlay`, `modal`, `modal-h/-b/-f`, `skills-editor`, `skills-md`, `skills-sub`,
  `skill-inv`; компонент `Select`. Цвета/шрифты — только токены `var(--…)` (гард `check:ds`).

### Важное проектное ограничение

Бэкенд `generate` пишет файл **в момент генерации**. Значит «превью ДО сохранения на сервере» без правки
бэкенда невозможно, а scope = только web. Поэтому превью трактуем как **экран-ревью уже созданного
скилла**: показываем, что получилось, и даём «Открыть» либо «Сгенерировать заново».

## 2. Решение

Превратить `CreateSkill` из одношаговой заглушки в **двухфазную модалку**:

```
Фаза 1 — Describe        Фаза 2 — Preview
──────────────────       ─────────────────────────────
[ описание (textarea) ]  Имя: <name> [invocable]
[ Account: <Select> ]    ┌─ превью SKILL.md (Markdown) ─┐
                         │ description + тело            │
[Cancel] [Create] ───►   └──────────────────────────────┘
                         [Regenerate]      [Open]
```

### 2.1 Фаза «Describe» (как сейчас, + блокировки)
- Поля: описание (textarea) + опциональный `Account` (`Select`, если есть `profiles`).
- Кнопка **Create** запускает `skillGenerate`. Во время запроса (`busy`):
  - кнопка показывает «Generating…», `disabled`;
  - textarea и `Select` тоже `disabled` (нельзя менять ввод на лету);
  - **Cancel** остаётся доступным (закрыть модалку).

### 2.2 AI-generate + валидация результата (клиентская)
По ответу `{ name, content }`:
1. **Валидация имени:** `name` соответствует слагу `^[A-Za-z0-9._-]+$`, не начинается с `-`, без `..`
   (зеркало бэкенда — на случай рассинхрона/будущих правок). Если нет — режим ошибки.
2. **Валидация структуры:** через `splitFrontmatter(content)` проверяем, что есть непустые `description`
   и `body` (frontmatter распознан, тело не пустое). Если нет — режим ошибки.
3. Успех → переход в фазу **Preview** с `{ name, content }` в состоянии.

> Бэкенд на кривой вывод уже отдаёт 422 (`skillGenerate` бросит исключение). Клиентская валидация
> страхует случай «200, но контент странный» и даёт единообразный inline-текст ошибки.

### 2.3 Фаза «Preview»
- Заголовок: имя скилла `name` + бейдж `invocable` (класс `skill-inv`), если `user_invocable: true`.
- Тело: `splitFrontmatter(content)` → `description` в `skills-sub` + `Markdown` тела в `skills-md`
  (как в правой панели просмотра — единый вид).
- Действия:
  - **Open** → `onCreated(name)` (закрывает модалку, перечитывает список, открывает скилл) — текущее
    «успешное» поведение, но теперь после осознанного просмотра.
  - **Regenerate** → вернуться в фазу Describe с сохранённым описанием и повторить `skillGenerate`
    (примечание: бэкенд создаст новый/перезапишет файл — это допустимо).

### 2.4 Состояния error (inline)
- Ошибку показываем **внутри модалки** (строка под полями/в превью), а не только тостом, чтобы не терять
  контекст и дать повтор. Текст: причина из исключения (`skillGenerate`) или сообщение валидации.
- Существующий `toast.error` можно сохранить как дубль либо убрать в пользу inline — выбрать на impl,
  не плодя двойных сообщений.

## 3. Вне scope (deferred)
- **Ручное создание «с нуля по имени»** (без AI) через `skillSave(name, template)` — технически возможно
  (бэкенд `writeSkill` создаёт новый `<name>/SKILL.md`), но это отдельная фича сверх брифа. Не делаем;
  фиксируем как возможное продолжение.
- Любые правки бэкенда / `src/core` (включая «настоящее» превью до записи на диск).

## 4. UI / копирайт
- Весь текст — **английский** (Create skill, Generating…, Open, Regenerate, Cancel, Account, сообщения об ошибках).
- Никаких хардкод-цветов/шрифтов — только существующие классы и токены `var(--…)` (иначе падает `check:ds`).
- Размерные инлайн-стили (`width/height`) допустимы (как уже используются).

## 5. Критерии приёмки
1. Из списка: **+ Create** → описать → **Create** → виден **превью** сгенерированного `SKILL.md`
   (имя + бейдж invocable + отрендеренное тело) → **Open** открывает скилл в правой панели.
2. Во время генерации видно состояние **busy**: «Generating…», поля и кнопка заблокированы.
3. При ошибке генерации/валидации показывается **понятная ошибка внутри модалки**, ввод сохранён,
   возможен повтор; модалка не закрывается сама.
4. Клиентская **валидация**: невалидное имя (не слаг) или отсутствие frontmatter `description`/тела →
   режим ошибки, а не «тихое» открытие битого скилла.
5. **Regenerate** из превью повторяет генерацию по тому же описанию.
6. `npm run check:ds` — **зелёный**; `tsc -b` (`npm run build`/`typecheck`) — **без ошибок**.
7. Существующие тесты **не сломаны** (`vitest`); бэкенд-тесты `skills/generate` не затронуты.

## 6. Риски
- **Дизайн-система:** новая разметка должна опираться на готовые классы/токены — главный источник падения `check:ds`.
- **Двойные сообщения об ошибке** (toast + inline) — выбрать один канал на impl.
- **Regenerate** перезаписывает файл на бэкенде — поведение приемлемо, но отметить в тексте, чтобы не удивляло.
- **Типы:** `content` из `skillGenerate` теперь используется — следить за `tsc -b`.
