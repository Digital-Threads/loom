# SDD — Раздел Knowledge: empty-state, recall/search на реальных данных, граф

Дата: 2026-06-17 · Задача журнала: `tj-77vc65ma82` · Класс: feature

## 1. Постановка

Раздел «Knowledge» (`web/src/components/Knowledge.tsx`) ищет по прошлым
рассуждениям из task-journal активного проекта: показывает, что уже **решено**
и что уже **отклонено**, чтобы команда/агент не повторяли пройденное.

Сейчас раздел работает наполовину. Нужно довести его до конца:

1. **Empty-state** — понятное пустое состояние до и после поиска.
2. **Recall/Search на реальных данных** — убедиться, что поиск читает журнал
   активного проекта и реально что-то находит; добавить в UI семантический
   поиск (`search`), которого там сейчас нет.
3. **Граф** — отрисовать problem→solution граф (эндпоинт и сборка уже есть, в UI
   не подключены).
4. Всё — по дизайн-системе (`npm run check:ds` в `web/`) и с тестами.

## 2. Текущее состояние (факты из кода)

- `web/src/components/Knowledge.tsx` (63 строки) — умеет только `recall`,
  рисует «⛔ Already rejected» и «✓ Already decided». Empty-state через
  `StateView kind="empty"` показывается **только** после поиска
  (`ran && decisions.length===0 && rejections.length===0`). До первого поиска —
  пустой экран без подсказки. Граф и search не используются.
- `web/src/api.ts`:
  - `recall(q)` → `{ hits, decisions, rejections }` (L233).
  - `knowledgeGraph(q)` → `{ nodes: unknown[]; edges: unknown[] }` (L238) — **в UI не вызывается**.
  - метода `search(q)` в клиенте **нет**, хотя серверный эндпоинт есть.
  - `RecallHit = { taskId; eventType; text; score }` (L362).
- Сервер `src/web/api.ts` L987–1001: `/api/knowledge/recall|search|graph`. Все
  три берут корень активного проекта
  (`resolveProjectRoot(projectActive().root ?? process.cwd())`) и вызывают
  `task-journal recall|ask --json` через `execFileSync` (пакет
  `@digital-threads/loom-knowledge`). То есть данные — реальный журнал активного
  проекта.
- `@digital-threads/loom-knowledge` `buildGraph(hits)`:
  `GraphNode = { id; kind: "decision"|"rejection"|"other"; taskId; label }`,
  `GraphEdge = { from; to }`. Узел на каждый хит; рёбра связывают хиты одной
  задачи по порядку (цепочка рассуждений).
- Дизайн-система: `web/package.json` → `check:ds` =
  `web/scripts/check-design-system.mjs` — запрещает хардкод цветов и шрифтов в
  компонентах; всё только из токенов `web/src/tokens/` через CSS-классы.
  Доступные токены: `--sev-info` (зелёный #43C46E), `--sev-warn` (амбер
  #E89B2C), `--sev-bug` (красный), `--mut`, `--fg-faint`, `--chip`, `--line`.
- Тестов на `Knowledge.tsx` в `test/web/` нет. Образец моков клиента —
  `test/web/client.test.ts` (`fakeFetch(routes)` подменяет `fetch`).
  В пакете knowledge есть `graph/recall/ask` тесты (логика сборки уже покрыта).

## 3. Объём работ

### 3.1 API-клиент (`web/src/api.ts`)
- Добавить метод `search(q)` →
  `getJson<{ hits: RecallHit[] }>("/api/knowledge/search?q=…")` рядом с `recall`.
- Типизировать граф: ввести в `web/src/api.ts` интерфейсы `GraphNode`,
  `GraphEdge`, `KnowledgeGraph` (зеркало пакета) и заменить
  `knowledgeGraph(q): {nodes: unknown[]; edges: unknown[]}` на возврат
  `KnowledgeGraph`.

### 3.2 Empty-state (`Knowledge.tsx`)
- **Начальное** состояние (ещё не искали, `!ran`): показать `StateView
  kind="empty"` с подсказкой как пользоваться (пример запроса), вместо пустого
  экрана.
- **После поиска без результатов**: оставить текущее
  `Nothing prior found for “…”`.
- **Загрузка**: на время запроса показывать `StateView kind="loading"`.

### 3.3 Граф (`Knowledge.tsx` + стили)
- После recall дополнительно запрашивать `client.knowledgeGraph(q)` (или
  строить из тех же hits — но эндпоинт уже даёт готовый граф, используем его).
- Отрисовать **без сторонней библиотеки**: сгруппировать узлы по `taskId`,
  внутри задачи показать цепочку узлов в порядке (рёбра = последовательность).
  Узлы — небольшие «карточки/чипы» с `label`, раскрашенные по `kind` через
  CSS-классы:
  - `decision` → класс на `--sev-info`,
  - `rejection` → класс на `--sev-warn`,
  - `other` → нейтральный (`--mut`/`--line`).
- Новые классы — в `web/src/styles.css` (или отдельный блок), цвета **только**
  токенами, чтобы пройти `check:ds`. Никаких inline-цветов.
- Граф показывать как отдельную секцию ниже списков «decided/rejected» (или под
  переключателем «список / граф» — решается на impl; минимально — секция ниже).
- Если граф пуст — секцию графа не рисовать.

### 3.4 Проверка на реальных данных
- Запустить хост, открыть раздел Knowledge для активного проекта, выполнить
  recall по реальному запросу, убедиться что возвращаются записи журнала и граф
  строится. Зафиксировать как evidence в журнале.
- Граничный случай: `task-journal` недоступен или журнал пуст → recall кидает /
  возвращает `[]`; UI должен показать empty-state, а не падать (toast.error уже
  есть на исключение).

## 5. Критерии приёмки

1. До первого поиска раздел показывает понятную подсказку (empty-state), не
   пустой экран.
2. Recall и search работают на реальном журнале активного проекта (проверено
   вживую, evidence в журнале).
3. Граф отрисовывается из реальных хитов: узлы decision/rejection/other
   раскрашены по дизайн-системе, цепочки сгруппированы по задаче; пустой граф не
   ломает экран.
4. `npm run test` зелёный; новые тесты на client.search/knowledgeGraph и на
   компонент проходят.
5. `npm --prefix web run check:ds` проходит без нарушений (нет хардкод-цветов).

## 6. Риски / открытые вопросы

- **Зависимость от CLI `task-journal`**: на машине без бинаря recall упадёт —
  обработать как empty/ошибку (toast), не как краш.
- **Глубина визуализации графа**: минимально — список цепочек по задачам;
  раскладку с рёбрами-линиями (SVG) делаем только если просто; решается на impl.
- **search vs recall в UI**: recall — «решено/отклонено», search — общий
  семантический поиск. Нужно ли отдельное поле/режим или общий запрос —
  уточняется на impl (минимально: общий запрос, search как дополнительная
  секция «похожее»).
- **check:ds на новые классы**: убедиться, что новые цвета добавлены в токены, а
  компонент использует только классы.
