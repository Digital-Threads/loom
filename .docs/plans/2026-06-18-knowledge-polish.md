# SDD — Полировка экрана Knowledge (граф recall)

- **Дата:** 2026-06-18
- **Класс задачи:** chore (полировка работающей фичи + удаление мёртвого кода)
- **Скоуп:** только `loom-host/web`
- **Файл задачи (task-journal):** `tj-wymq34gxsp`

## 1. Цель

Довести до ума уже работающий экран «Knowledge» (`web/src/components/Knowledge.tsx`) —
раздел, где по запросу ищут прошлые рассуждения проекта: что уже **решали**, что уже
**отвергли**, граф цепочек рассуждений и похожие записи. Поиск (recall + search) и его
поведение должны остаться без изменений; меняется только визуальная подача и убирается
мёртвый код.

## 2. Контекст и текущее состояние

Экран реализован в коммите `5c3cbfb`. Сейчас он уже умеет:

- Поле ввода + кнопка **Recall** (поиск по Enter и по клику), защита от повторного входа.
- Параллельный вызов `client.recall(q)` и `client.search(q)` через `Promise.allSettled` —
  падение одного не роняет другой, на ошибку показывается toast.
- Секции: **⛔ Already rejected**, **✓ Already decided**, **🕸 Reasoning graph**,
  **≈ Similar prior reasoning** (с дедупликацией: похожие, уже показанные как хит, не дублируются).
- Граф строится на клиенте из хитов (`chainsFromHits`, Knowledge.tsx:21-28): группировка по
  `taskId`, внутри задачи узлы соединяются стрелками `→` (рёбра = цепочка по задаче).
- Узлы графа имеют класс по виду — `.kg-node.decision` / `.rejection` / `.other`
  (вид определяется `kindOf`, Knowledge.tsx:11-15).
- Состояния через общий `StateView`: `loading`, начальная подсказка (до первого поиска),
  «ничего не найдено».

### Что не так / что доводим

1. **Мёртвая заглушка графа.** В `api.ts` живёт неиспользуемый клиентский метод
   `knowledgeGraph` (api.ts:248-249) и типы `KnowledgeGraph` / `GraphNode` / `GraphEdge`
   (api.ts:382-384). Это старый «серверный граф» (`/api/knowledge/graph`), заменённый
   построением цепочек на клиенте. Компонент его не вызывает; тест `Knowledge.test.tsx`
   мокает его впустую.
2. **Цвета узлов графа.** Сейчас цвета уже из токенов DS (`styles.css:207-209`):
   `decision → --sev-info`, `rejection → --sev-warn`, `other → --line/--mut`. Требование —
   подтвердить, что цвета берутся **только** из токенов DS, и выверить читаемость.
3. **Плотность списка хитов.** Строки `.kv` (styles.css:199) и заголовки секций — сделать
   списки компактнее без потери читаемости.
4. **Начальное / пустое состояние.** Тексты есть, нужно выверить формулировки (UI английский)
   и единообразие.

## 4. Требования

### Функциональные

- **FR-1.** recall + search работают как раньше: параллельно, независимо, ошибки → toast.
- **FR-2.** Граф: узлы трёх видов (decision/rejection/other), внутри одной задачи соединены
  стрелками `→`; группы — по `taskId`. Поведение сохраняется.
- **FR-3.** Удалить мёртвую заглушку `knowledgeGraph` и связанные неиспользуемые типы;
  компонент по-прежнему строит граф из хитов на клиенте.
- **FR-4.** Состояния: до первого поиска — подсказка; пусто после поиска — «ничего не найдено»;
  во время поиска — loading. Тексты на английском.

### Нефункциональные / ограничения

- **NFR-1.** Цвета и шрифты — **только** через токены DS (`var(--…)`); никаких новых hex
  или inline-цветов в `.tsx`. `npm run check:ds` (`scripts/check-design-system.mjs`) — зелёный.
- **NFR-2.** `tsc -b` без ошибок.
- **NFR-3.** `vitest run` зелёный (тесты не ломаем; мок синхронизируем под удаление API).
- **NFR-4.** Изменения только в `loom-host/web`.
- **NFR-5.** UI на английском.

## 5. Дизайн решения

### 5.1. Удаление заглушки (api.ts)

Удалить из `api.ts`:
- метод клиента `knowledgeGraph` (248-249);
- типы `KnowledgeGraph`, `GraphNode`, `GraphEdge` (382-384).

Оставить `RecallHit` и `GraphNodeKind`. Так как `chainsFromHits` строит цепочки сам, никакой
функциональности не теряем. После удаления — обновить `Knowledge.test.tsx`: убрать строку
мока `knowledgeGraph` и импорт типа `KnowledgeGraph` (мок кастуется `as unknown as LoomClient`,
поэтому лишнее свойство компиляцию не ломает, но чистим для соответствия).

### 5.2. Граф — цвета только токенами

Подтвердить/сохранить токенную раскраску (`styles.css:207-209`):
- `decision` → `--sev-info`;
- `rejection` → `--sev-warn`;
- `other` → `--line` (рамка) + `--mut` (текст).

Любая правка узлов — только через классы и `var(--…)`. Inline-цветов в `Knowledge.tsx` не вводим.

### 5.3. Плотность списка хитов

Поднять плотность строк `.kv` (вертикальные отступы) и при необходимости отступы заголовков
секций `.panel h2`, чтобы длинные списки читались компактнее. Только размеры/отступы, цвета не трогаем.

### 5.4. Состояния

Сохранить три состояния через `StateView`. Выверить тексты (английский), при правках —
формулировки короткие и единообразные.

## 6. Критерии приёмки

1. `cd web && npm run check:ds` → «✓ Design-system guard…» (зелёный).
2. `cd web && npx tsc -b` → без ошибок.
3. `cd web && npm run test` (vitest) → все тесты зелёные, включая `Knowledge.test.tsx`.
4. `client.knowledgeGraph` и типы `KnowledgeGraph`/`GraphNode`/`GraphEdge` отсутствуют в `web/src`.
5. Граф рисует узлы decision/rejection/other с токенными цветами и рёбрами-цепочками по задаче.
6. Видны все четыре состояния (initial / loading / nothing-found / результаты), тексты на английском.
7. Изменения не выходят за пределы `loom-host/web`.

## 7. Риски и открытые вопросы

- **R-1.** Удаление `knowledgeGraph` затрагивает `api.ts` и мок теста — держать `tsc -b` и
  vitest зелёными. Митигировать: синхронно обновить тест.
- **R-2.** «Плотность» субъективна — критерий: компактнее, но читаемо; цвета только токенами.
- **R-3.** Семантика цвета (`decision` зелёный, `rejection` янтарный) выглядит логично —
  менять токены, скорее всего, не нужно; если менять, то только на существующие токены DS.
