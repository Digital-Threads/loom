# SDD — Onboarding-визард первого запуска (D2.2)

**Тип:** доводка существующего компонента (не новая фича).
**Дата:** 2026-06-18.
**Класс:** feature. **Маршрут:** analysis → brainstorm → **spec** → impl → review → qa → pr → done.

---

## 1. Цель (одним предложением)

Первый запуск Loom на чистой машине должен провести человека через настройку
**без терминала** — показать состояние среды, дать добавить подписку и первый
проект, и по завершении открыть рабочую доску.

## 2. Проблема (что сейчас)

Экран первого запуска `web/src/components/Onboarding.tsx` (62 строки) уже есть,
но «сырой»:

- Показывается, когда в системе нет ни одного проекта
  (`web/src/App.tsx:60-86`, гейт `onboard = projects.length === 0`).
- Делает только два шага: ввод имени aimux-профиля → `addSubscription`,
  и путь к проекту → `addProject` → `onDone`.
- Среду **не проверяет**: показывает лишь число профилей
  (`workspace().subscriptions.length`), без понимания «есть ли вообще
  рабочий AI-CLI» и «здоров ли профиль».
- Состояния загрузки/ошибки минимальны: `subs ?? "…"` и одна строка `msg`.
  Нет явного «пусто», нет внятной ошибки.
- Тестов на компонент нет.

## 4. Объём работ (по скоупу D2.2)

### 4.1. Проверка среды (скоуп #1) — нужна тонкая бэкенд-ручка

**Решение:** добавить read-only ручку `GET /api/doctor`, оборачивающую
существующую `checkPrerequisites()`, и метод клиента `doctor()`.

- Ответ: `{ ok: boolean, tools: { name, found, hint }[], missing: string[] }`
  (ровно `PrereqReport` из `prereqs.ts`).
- Фронт показывает понятный статус каждого инструмента: «найден / не найден +
  подсказка где взять». Особо — статус AI-CLI (`claude`).

**Альтернатива (отклонена):** обойтись только `accountsHealth()`. Отклонено —
он показывает здоровье aimux-профиля, но не наличие AI-CLI в PATH, а скоуп #1
требует именно «есть ли AI-CLI в PATH». Тонкая обёртка над уже существующей
функцией ядра — это «бэкенд-ручка» из скоупа, а не новая фича.

### 4.2. Подписка, если нет здоровых профилей (скоуп #2)

- Считать «здоровые профили» через то же правило, что в `Accounts.tsx`
  (`broken+missing+conflicts === 0`), по данным `workspace().health`.
- Если здоровых нет — показать шаг «Connect an AI subscription» с вводом имени
  профиля → `addSubscription`. После успеха — перечитать `workspace()` и
  обновить статус. Ошибку из ответа (`{ ok:false, error }`) показывать явно.
- Если здоровый профиль уже есть — шаг отмечается выполненным (галочка), не
  блокирует переход дальше.

### 4.3. Первый проект, если проектов нет (скоуп #3)

- Существующий шаг сохраняется: путь к репозиторию + `DirectoryPicker`
  («Browse…») → `addProject(root)` → `onDone()`.
- Добавить состояние ошибки добавления проекта (сейчас ошибка глотается).

### 4.4. Рекомендованные слои (скоуп #4) — ПРОПУСКАЕМ, отмечено

Механизм каталога установки (`buildCatalog`, L11.2) **в master через web-API не
выставлен**. По правилу задачи («если механизм есть; если нет в master —
пропустить и отметить») — пункт 4 в этой доводке не реализуем; выставление
каталога через API выходит за рамки «доводки онбординга». Отмечено здесь явно.

### 4.5. Состояния и завершение (скоуп #5)

- **loading** — пока грузятся `doctor()` + `workspace()` (статус «Checking…»).
- **error** — если запрос упал, показать сообщение и кнопку «Retry», не белый
  экран.
- **empty / готово** — для каждого шага явный статус (нужно действие / готово).
- **done** — после добавления проекта вызвать `onDone()`; `App.tsx` снимает
  гейт и перезагружает → пользователь на доске. Гейт `App.tsx` не меняем.

## 5. Дизайн-система и UI

- UI на английском.
- Только DS: никаких инлайн-цветов и хардкод-hex; стили через существующие
  классы + токены. `npm run check:ds` остаётся зелёным.
- Новых селектов/чекбоксов скоуп не требует (слои пропущены). Если понадобится
  выбор — только через DS `Select`/DS-чекбокс.
- Тексты статусов короткие и понятные: «AI CLI found / not found», «Profile:
  healthy / N issues / none yet».

## 6. Тестирование (TDD, где тестируемо)

- **Backend:** в `test/web/api.test.ts` — тест `GET /api/doctor`: возвращает
  `PrereqReport` (мок-runner, как в тестах prereqs/install).
- **Client:** в `test/web/client.test.ts` — `doctor()` дергает `/api/doctor`.
- **Component:** новый `web/src/components/Onboarding.test.tsx` (паттерн
  `mkClient`): 
  - loading → виден «Checking…»;
  - нет здоровых профилей → виден шаг подписки;
  - есть здоровый профиль → шаг подписки отмечен готовым;
  - ошибка `doctor()`/`workspace()` → видно сообщение + Retry;
  - успешный `addProject` → вызван `onDone`.
- Не ломать существующие тесты: `npm test` зелёный.

## 7. Что НЕ трогаем

- Гейт онбординга в `App.tsx` (логика «нет проектов → онбординг»).
- Поведение `addSubscription`/`addProject`/`accountsHealth` на бэкенде.
- Каталог слоёв и любой новый API под него (вне скоупа доводки).
- Дизайн-токены.

## 8. Критерии приёмки

1. На чистой машине (нет CLI / нет профилей / нет проектов) онбординг ясно
   показывает: статус среды (AI-CLI), статус профиля, шаг проекта.
2. Можно добавить подписку и первый проект, не открывая терминал.
3. Загрузка/ошибка/пусто/готово — у каждого шага явный статус; ошибка не белый
   экран, есть Retry.
4. По добавлении проекта — переход на доску (`onDone`).
5. `npm run check:ds` зелёный; `npm test` зелёный (старые тесты целы, новые
   проходят).

## 9. Открытые вопросы

- Имя ручки: `GET /api/doctor` vs `GET /api/prereqs` — выбрать на impl
  (предлагается `/api/doctor`, т.к. в ядре каталог называется `doctor`).
- Нужен ли в онбординге показ aimux-`auth` (OAuth-флоу `authStart/authCode`)
  для «здоровья» нового профиля, или достаточно `addSubscription`. Предложение:
  в рамках доводки ограничиться `addSubscription` + показом здоровья; полный
  OAuth-флоу уже живёт в `Accounts.tsx`/`AuthModal.tsx` и доступен после
  онбординга.
