# План (R&D / DAG) — Skills: флоу создания + AI-generate

SDD: `.docs/specs/2026-06-18-skills-create-flow.md`. Scope — только `web/src/components/Skills.tsx`
(компонент `CreateSkill`). Бэкенд (`POST /api/skills/generate`) и API-клиент (`skillGenerate → {name, content}`)
уже готовы — не трогаем. Весь план — фронтенд, один файл.

## Граф зависимостей
```
S1 (state-машина модалки)
   └─► S2 (фаза Describe + busy-блокировки)
          └─► S3 (валидация результата generate)
                 └─► S4 (фаза Preview + Open/Regenerate)
                        └─► S5 (inline error)
                               └─► S6 (верификация: check:ds + tsc -b + vitest)
```
S1→S5 — последовательные правки одного компонента; S6 — финальная проверка.

---

## S1 — Состояние/каркас модалки (Describe ↔ Preview)
- **Что:** ввести в `CreateSkill` фазовое состояние: `phase: "describe" | "preview"` и хранилище
  результата `result: { name, content } | null`. Подготовить заготовку рендера обеих фаз.
- **Файлы:** `web/src/components/Skills.tsx` (только `CreateSkill`).
- **Готово:** компонент компилируется (`tsc -b`), поведение пока эквивалентно текущему (фаза describe),
  новые поля состояния заведены и типизированы; внешний контракт `onClose/onCreated/profiles` не изменён.

## S2 — Фаза Describe + блокировки busy
- **Что:** оставить поля (описание + `Account` Select); при `busy` (идёт `skillGenerate`) блокировать
  textarea и `Select`, кнопка показывает «Generating…» и `disabled`; **Cancel** доступен.
- **Файлы:** `web/src/components/Skills.tsx`.
- **Готово:** во время генерации поля недоступны для ввода, кнопка Create заблокирована и показывает busy-текст;
  тексты английские; новых цветов/шрифтов нет (классы/токены — `check:ds` зелёный локально на глаз).

## S3 — Валидация результата generate (клиентская)
- **Что:** по ответу `{name, content}` проверить:
  (1) имя — слаг `^[A-Za-z0-9._-]+$`, не с `-`, без `..` (зеркало бэкенд-`safeName`);
  (2) структура — `splitFrontmatter(content)` даёт непустые `description` и `body`.
  Невалидно → перейти в режим ошибки (S5), не открывать скилл. Валидно → перейти в Preview (S4).
- **Файлы:** `web/src/components/Skills.tsx` (можно вынести маленький helper `validateSkill(name, content)`
  рядом со `splitFrontmatter`, переиспользуя последний).
- **Готово:** для валидного контента — переход в Preview; для битого (нет frontmatter / плохое имя) —
  режим ошибки; `tsc -b` чистый.

## S4 — Фаза Preview + действия Open / Regenerate
- **Что:** показать сгенерированный скилл: имя + бейдж `invocable` (`skill-inv`), описание (`skills-sub`)
  и тело через `Markdown` (`skills-md`) — как в правой панели просмотра. Кнопки:
  **Open** → `onCreated(name)`; **Regenerate** → вернуться в Describe (описание сохранено) и повторить генерацию.
- **Файлы:** `web/src/components/Skills.tsx`.
- **Готово:** после успешной генерации виден превью с корректным телом/именем/бейджем; **Open** открывает
  скилл в правой панели (текущее поведение `onCreated`); **Regenerate** повторяет `skillGenerate` по тому же описанию.

## S5 — Inline-ошибка (вместо/в дополнение к toast)
- **Что:** хранить `error: string | null`; показывать понятную строку ошибки **внутри модалки**
  (причина из исключения `skillGenerate` или сообщение валидации S3); ввод сохранён, доступен повтор;
  модалка сама не закрывается. Выбрать один канал сообщений (inline), убрав дублирующий `toast.error`
  либо оставив один — без двойных сообщений.
- **Файлы:** `web/src/components/Skills.tsx`.
- **Готово:** при ошибке генерации/валидации текст виден в модалке, кнопка снова активна, повтор работает;
  нет двойного уведомления.

## S6 — Верификация
- **Что:** прогнать гард дизайн-системы и сборку типов и тесты.
- **Команды (в `web/`):** `npm run check:ds`; `npm run build` (или `npm run typecheck` для быстрого `tsc`);
  `npm run test` (vitest).
- **Готово:** `check:ds` — зелёный; `tsc -b` — без ошибок; `vitest` — без новых падений (существующие
  тесты не сломаны); бэкенд-тесты `skills/generate` не затронуты.

---

## Вне scope (deferred)
- Ручное создание скилла «с нуля по имени» через `skillSave` — отдельная фича, не в этом плане.
- Любые правки бэкенда / `src/core` / «настоящего» превью до записи на диск.

## Заметки/риски
- Главный риск — `check:ds`: новая разметка только на готовых классах/токенах, без хардкод-цветов.
- `Regenerate` перезаписывает файл на бэкенде (generate пишет сразу) — поведение приемлемо, отметить в UI-тексте при impl.
- Тесты для `Skills` отсутствуют; добавление web-теста — опционально, бриф требует лишь «не ломать тесты».
