# SDD — Полировка всех разделов меню к релизу

- **Дата:** 2026-06-18
- **Класс:** chore (релизная доводка, без нового функционала)
- **Грунт (только эти области):** `web/src/components/*`, `web/src/styles.css`, `web/src/tokens/*`
- **Ветка:** `loom/t-981577ba` (перезапуск начисто от текущего `master`)

---

## 1. Цель

Довести до релизного качества **каждый** из 14 разделов меню. Это не новая
функциональность, а устранение шероховатостей: чтобы каждый раздел открывался
без ошибок, основные действия работали, а пустые/загрузочные/ошибочные состояния
были понятными — всё строго по дизайн-системе Loom, на английском, с сохранённой
доступностью.

## 3. Гард дизайн-системы (`npm run check:ds`)

Скрипт `web/scripts/check-design-system.mjs` (запуск: `npm run check:ds` в `web/`,
либо `npm run check:ds` из корня) обходит `web/src` (кроме `tokens/` и `*.test.*`)
и **проваливает сборку**, если найдёт:

1. Захардкоженный hex-цвет, кроме разрешённых `#fff`, `#ffffff`, `#ff8a80`, `#6fe3a0`.
2. «Старую палитру» (запрещённые литералы вроде `#5b9bff`, `#7c5cff`, `#0e1116`,
   соответствующие `rgba(...)`).
3. Инлайн-цвет в компоненте: `color:`/`background:` со значением `#`/`rgb` прямо в `.tsx`.
4. `font-family:` без `var(--font-…)` / `--sans` / `--mono` / `--display`.

**Вывод:** любые правки цвета/шрифта — только через токены и классы; в `.tsx`
никаких инлайновых цветов.

## 4. Скоуп

### В скоупе
- Доводка 14 существующих разделов по четырём осям (см. критерии приёмки).
- Правки только в `web/src/components/*`, `web/src/styles.css`, `web/src/tokens/*`.
- Тесты (Vitest + Testing Library) там, где поведение тестируемо.

### Вне скоупа (NON-GOALS)
- Новые разделы, новые действия, новые поля/настройки.
- Изменения серверного API / бизнес-логики бэкенда.
- Редизайн дизайн-системы (токены меняем только если без этого не закрыть баг).
- Рефакторинг «заодно» в файлах вне грунта или не связанный с конкретной шероховатостью.

## 5. Критерии приёмки (для каждого раздела)

Раздел считается «отполированным», когда выполнены все четыре оси:

- **R — Рендер.** Открывается без ошибок в консоли и без падений; начальный
  рендер (до загрузки данных) корректен.
- **A — Действия.** Основные действия раздела работают (клик/ввод/сохранение
  дают ожидаемый результат и обратную связь — тост/обновление состояния).
- **S — Состояния.** Loading / empty / error отображаются через `StateView`
  (или эквивалентный DS-стиль), информативны и на английском; ошибки не «глотаются»
  молча, пустое состояние объясняет, что делать.
- **D — Дизайн-система и доступность.**
  - Нет нативных «голых» контролов: каждый `<select>` — через `Select`, чекбоксы —
    через DS-класс.
  - `npm run check:ds` зелёный (нет хардкода цветов/шрифтов, нет инлайн-цветов).
  - UI на английском.
  - Доступность сохранена: корректные роли, видимый фокус, `aria-*`
    (`aria-current`, `aria-label`, `aria-hidden` на декоративных иконках),
    модалки с ловушкой фокуса (`Modal.tsx`).

## 6. План работ

1. **Инвентаризация (read-only).** Пройти 14 компонентов через token-pilot
   (`smart_read`/`read_symbol`), составить чек-лист конкретных шероховатостей по
   осям R/A/S/D. Зафиксировать находки.
2. **Базовая проверка.** Прогнать `npm run test` и `npm run check:ds` от текущего
   `master` — зафиксировать «зелёный старт», чтобы не путать новые регрессии со
   старыми.
3. **Доводка по разделам, хирургически.** Для каждого раздела — минимальные
   правки под найденные пункты. Где поведение тестируемо: сначала падающий тест,
   потом фикс (TDD). Коммит на каждый зелёный шаг.
4. **Сквозные правки.** Унифицировать состояния через `StateView`, заменить
   оставшиеся нативные контролы на DS-обёртки — только там, где это реальная
   шероховатость, а не косметика ради косметики.
5. **Финальная проверка.** `npm run check:ds` зелёный, `npm run test` зелёный,
   ручной проход всех 14 разделов (рендер + основное действие + состояния).

## 7. Проверка/доказательство готовности

- **Обязательно:** `npm run check:ds` → `✓ Design-system guard…`.
- **Обязательно:** `npm run test` (Vitest) — все существующие тесты зелёные,
  новые добавленные тесты проходят. Существующие тесты не ломаем.
- **Ручной чек-лист 14 разделов** по осям R/A/S/D (см. §5) — каждый отмечен.
- Diff остаётся в пределах грунта (§ грунт); каждая изменённая строка ведёт к
  конкретной шероховатости.

## 8. Риски и открытые вопросы

- **«Голый» `<select>` внутри `Select.tsx`.** DS-обёртка сама использует нативный
  `<select>` под стилизацией — это и есть DS-контрол. Не путать с прямыми
  `<select>` в разделах: заменять надо именно последние.
- **Объём «шероховатостей» заранее не известен.** Реальный список рождается на
  шаге инвентаризации; SDD задаёт критерии, а не фиксированный перечень багов.
- **Тестируемость состояний.** Часть error/empty веток завязана на ответы API —
  тестируем через мок-клиент (есть пример: `Board.test.tsx`, `Knowledge.test.tsx`).
- **Токены трогаем в крайнем случае.** Если шероховатость требует нового значения
  цвета/отступа — добавляем токен в `tokens/*`, не хардкодим в компоненте.
- **Окружение task-journal недоступно** (MCP не находит файл проекта) — ход работы
  фиксируется в этом SDD и коммитах вместо журнала.

## 9. Определение «Готово»

Все 14 разделов проходят R/A/S/D; `check:ds` и тесты зелёные; изменения
хирургические и в пределах грунта; новый функционал не добавлен.
