# SDD — Полировка форм Settings / Projects / Onboarding (loom-host/web)

- **Дата:** 2026-06-18
- **Задача (journal):** tj-fgbp6mwpk5
- **Класс:** feature (UI-полировка)
- **Область:** только пакет `loom-host/web`. Другие пакеты не трогаем.

## 1. Цель

Привести три формы веб-интерфейса к единому, аккуратному и «защищённому от ошибок» виду:

1. Валидация ввода — нельзя отправить пустое/некорректное значение; причина видна у поля.
2. Состояния `loading / error / success` — единообразны во всех трёх формах.
3. Плотность вёрстки — компактнее и ровнее, без лишнего «воздуха».
4. Убрать `placeholder` в Projects (заменить на видимую подпись/`aria-label`).

## 2. Границы (что НЕ делаем)

- Не меняем серверные API и их контракты (`web/src/api.ts` — только используем).
- Не трогаем `placeholder` в `Onboarding.tsx` — на них завязан тест
  (`getByPlaceholderText("/path/to/repo")`, `/profile name/`).
- Не вводим новые цвета/шрифты хардкодом — только токены (`var(--…)`), иначе `check:ds` падает.
- Не добавляем новых зависимостей и абстракций. Минимум кода.

## 3. Затрагиваемые файлы

| Файл | Что меняем |
| --- | --- |
| `web/src/components/Projects.tsx` | убрать placeholder, валидация поля пути, состояния, плотность |
| `web/src/components/Settings.tsx` | валидация cost cap, единый success/loading, плотность |
| `web/src/components/Onboarding.tsx` | единый стиль ошибок/успеха полей, плотность (placeholder не трогаем) |
| `web/src/styles.css` | при необходимости — маленький класс ошибки поля + утилиты плотности (через токены) |

Вспомогательное (используем как есть): `StateView` (loading/error/empty), `toast` (success/error).

## 4. Контракты API (факт из `createClient`, web/src/api.ts)

- `addProject(root)` → `{project}` **или throw** (поля `error` нет) → ошибка ловится `try/catch`.
- `saveSetting(key, value)` → `{ok}`.
- `addSubscription(name)` → `{ok, error?}`.
- `removeProject(id)` → `{ok, error?}`.
- `setActiveProject(id)` → `active`.

Вывод: в Projects/Onboarding ошибка добавления проекта приходит только как исключение (текущий
паттерн сохраняем), у подписки — через `r.ok/r.error`.

## 5. Дизайн-решения

### 5.1 Состояние success — единообразно через `toast`
Уже используется в Projects/Settings. Onboarding success-сообщения (`subMsg`) приводим к тому же
тону. **Не** добавляем отдельные inline success-баннеры у полей — это дублирует toast и плодит
состояния.

> Отвергнуто: inline success-баннер у каждого поля — лишняя сложность без пользы.

### 5.2 Ошибки и валидация — на двух уровнях
- **Уровень формы** (не удалось загрузить данные): остаётся через `StateView kind="error"` /
  блок `.state-err` (как в Onboarding `projErr`).
- **Уровень поля** (валидация ввода): короткое сообщение под полем + кнопка `disabled`, пока ввод
  невалиден. Для текста ошибки поля вводим маленький класс (см. 5.5), цвет — `var(--bad)`.

### 5.3 Конкретная валидация по формам
- **Projects — путь проекта:** кнопка `Add project` уже `disabled` при пустом; добавить тот же
  паттерн, что в Onboarding (`disabled={busy || !root.trim()}`), и не сбрасывать введённое при
  ошибке (сейчас `setRoot("")` только при успехе — оставить так). Глубокую проверку пути не
  делаем — её делает backend.
- **Settings — cost cap:** сейчас сохраняется в `onBlur` как `Number(e.target.value) || 0` без
  защиты от отрицательных. Добавить: отрицательные/`NaN` → `0`, поле `min={0}` уже есть; при
  «починке» значения отразить это в поле (обновить отображаемое значение), success-toast «Saved».
- **Onboarding — profile / root:** кнопки уже `disabled` при пустом вводе — сохранить, привести
  сообщения об ошибке к единому виду.

### 5.4 Убрать placeholder в Projects
`Projects.tsx:52` — у `<input className="inp" placeholder="/path/to/repo" …>` убрать атрибут
`placeholder`. Чтобы поле осталось понятным и доступным: добавить видимую подпись (label) над/слева
от поля **или** `aria-label="Project repository path"`. Подпись на английском.

### 5.5 Плотность вёрстки
- Править существующие классы (`.kv`, `.acct-add`, `.acct-hint`, `.row`) и числовые inline-отступы
  (`margin/padding` — `check:ds` их разрешает; запрещены только inline `color/background` и хардкод
  цветов).
- Свести разнобой отступов между шагами Onboarding (`marginTop: 18`) и `.kv` Settings к согласованным
  значениям. Без визуальной перестройки — только подтянуть интервалы.
- При необходимости добавить в `styles.css` крошечный класс ошибки поля, например:
  `.field-err{margin-top:4px;font-size:12px;color:var(--bad)}` (цвет — токен, `check:ds` зелёный).

## 6. Ограничения / инварианты

- `npm run check:ds` (`node scripts/check-design-system.mjs`) — **зелёный**: ни одного хардкод-hex
  (кроме разрешённого набора), ни одного inline `color/background` в `.tsx`, шрифты только `var(--font-…)`.
- UI — **английский** (новые подписи и сообщения валидации тоже).
- `tsc -b` — без ошибок.
- `npm test` (vitest) — **не ломать**; критично сохранить placeholder'ы Onboarding.

## 7. Критерии приёмки

1. В Projects у поля пути нет атрибута `placeholder`, но поле имеет видимую подпись или `aria-label`;
   кнопка добавления заблокирована при пустом вводе.
2. В Settings нельзя сохранить отрицательный/нечисловой cost cap — приводится к `0` и это видно в поле;
   при сохранении — toast «Saved».
3. Все три формы показывают понятные `loading` и `error` (с возможностью повтора там, где он уже есть),
   а успешные действия — через `toast`.
4. Формы визуально плотнее и с согласованными отступами; ничего не «разъехалось».
5. `check:ds`, `tsc -b`, `vitest` — зелёные.

## 8. План проверки (QA)

- `cd web && npm run check:ds` → ✓.
- `cd web && npx tsc -b` (или `npm run typecheck`) → без ошибок.
- `cd web && npm test` → все тесты проходят, включая `Onboarding.test.tsx`.
- Ручная проверка форм в `npm run dev`: пустые поля, отрицательный cost cap, ошибка добавления проекта.

## 9. Риски

- **Тест Onboarding** завязан на placeholder'ы и тексты — менять их формулировки осторожно.
- **`check:ds`** легко уронить случайным inline-цветом — все цвета только через токены/классы.
- Изменение плотности может затронуть соседние экраны через общие классы (`.kv`, `.row`) — проверить,
  что правки локальны или согласованы глобально.
