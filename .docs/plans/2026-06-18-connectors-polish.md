# SDD — Полировка Connectors (import-flow + статусы подключения)

- **Дата:** 2026-06-18
- **Область:** только `loom-host/web`
- **Файл изменений:** `web/src/components/Connectors.tsx` (единственный)
- **Задача (journal):** tj-0vtac596n7
- **Класс:** chore (полировка двух заглушек, без новой доменной логики)

## 1. Контекст и проблема

Экран **Connectors** управляет MCP-серверами (список / добавить / включить-выключить / проверить / удалить) и умеет импортировать задачи из beads. Сейчас два места работают, но выглядят сыро («заглушки»):

1. **Импорт из beads.** Кнопка `Import from beads` зовёт `client.importTracker()` → `{ created: number }`. Результат пишется одной серой строкой `imported N`. Нет индикации «идёт импорт», нет внятного финального итога, ошибка показывается только во всплывающем toast и не остаётся на экране.
2. **Статусы проверки коннекторов.** Кнопка `Test` зовёт `client.mcpTest(id)` → `{ ok, error? }`. Результат кладётся сырой строкой в ячейку команды как `· ok` / `· <текст ошибки>`. Это невнятно: нельзя отличить «не проверяли» от «проверяется» от «доступен/недоступен».

Состояния пусто/ошибка/загрузка для **списка серверов** уже есть (через общий `StateView`) — их менять не нужно; новые куски должны вести себя так же единообразно.

## 2. Цели и не-цели

**Цели**
- Импорт из beads показывает понятный поток: запуск → идёт импорт → итог (сколько создано) либо ошибка, и итог/ошибка остаются на экране.
- У каждого сервера понятный статус проверки в виде чипа: «не проверялся» / «проверяется…» / «доступен» / «недоступен» (англоязычные подписи).
- Состояния пусто/ошибка/загрузка единообразны (переиспользуем `StateView` и существующие чипы).

**Не-цели**
- Не меняем backend и сигнатуры API (`importTracker`, `mcpTest` остаются как есть).
- Не добавляем настоящий многошаговый прогресс импорта (бэкенд отдаёт только `{ created }` — «шаги» это UX-индикация на фронте).
- Не трогаем добавление/вкл-выкл/удаление серверов и список целиком.
- Не добавляем новый CSS/токены.

## 3. Ограничения (обязательны к соблюдению)

- **Только `loom-host/web`.**
- **`npm run check:ds` зелёный.** Гард `web/scripts/check-design-system.mjs` запрещает в `.tsx` любой inline-цвет (`color`/`background`) и hardcoded hex вне белого списка. → Все цвета только через классы-чипы. Inline нецветовые стили (`gap`, `marginLeft`) допустимы.
- **UI на английском** — все новые подписи.

- **Тесты не ломать** (`npm test` = `vitest run`).

## 4. Дизайн решения

Изменения локализованы в `Connectors.tsx`. Опираемся на уже существующие в `web/src/styles.css` классы чипов — **новый CSS не нужен**:

| Класс | Цвет (токен) | Назначение здесь |
|-------|--------------|------------------|
| `.chip` | нейтральный (`--mut`) | «не проверялся» / нейтральный счётчик |
| `.chip.ok` | зелёный (`--done`) | «доступен» / успешный итог импорта |
| `.chip.warn` | янтарный (`--run`) | «проверяется…» / «идёт импорт…» |
| `.chip.bad` | красный (`--fail`) | «недоступен» / ошибка импорта |
| `.chip.sm` | — | мелкий размер, по желанию |

### 4.1. Модель состояния (внутри компонента)

Сейчас всё свалено в `status: Record<string, string>` (и сервера, и импорт). Заменяем на типизированные состояния:

- **Статус проверки сервера** — отдельная мапа:
  ```ts
  type TestState =
    | { kind: "idle" }                 // не проверялся (можно просто не хранить ключ)
    | { kind: "checking" }             // проверяется…
    | { kind: "ok" }                   // доступен
    | { kind: "fail"; error?: string };// недоступен (+ причина, если есть)
  const [tests, setTests] = useState<Record<string, TestState>>({});
  ```
  Отсутствие ключа = «idle / не проверялся» (чип не показываем или показываем нейтральный по желанию).

- **Статус импорта** — отдельное состояние (не мешать с серверами):
  ```ts
  type ImportState =
    | { kind: "idle" }
    | { kind: "running" }              // идёт импорт…
    | { kind: "done"; created: number }// импортировано N
    | { kind: "error"; message: string };
  const [imp, setImp] = useState<ImportState>({ kind: "idle" });
  ```

> Тип `status: Record<string,string>` убираем как избыточный (это и есть «заглушка»).

### 4.2. Поток импорта из beads

- Клик по `Import from beads`: ставим `imp = { kind: "running" }`, кнопку дизейблим на время выполнения.
- Успех: `imp = { kind: "done", created: r.created }` + toast success.
- Ошибка: `imp = { kind: "error", message: String(e) }` + toast error.
- **Рендер итога рядом с кнопкой** (а не одной серой строкой):
  - `running` → чип `warn` с текстом `Importing…`;
  - `done` → чип `ok` с текстом `Imported {created}` (при `created === 0` — нейтральный текст `Nothing new to import`);
  - `error` → используем `StateView kind="error" msg={message}` или чип `bad` с коротким текстом — итог остаётся на экране, не только в toast.

### 4.3. Статусы проверки серверов

- Клик по `Test(id)`: `tests[id] = { kind: "checking" }`, далее по ответу `ok`/`fail`.
- **Рендер чипа статуса** в строке сервера (рядом с `on`/`off` или в отдельной ячейке/в ячейке команды вместо `· ok`):
  - нет записи → ничего (или нейтральный `Untested`);
  - `checking` → чип `warn` `Checking…`;
  - `ok` → чип `ok` `Reachable`;
  - `fail` → чип `bad` `Unreachable` (тултип/подпись с `error`, если есть).
- Кнопку `Test` дизейблим, пока `checking` для этого сервера.

### 4.4. Состояния пусто/ошибка/загрузка

- Список серверов: оставляем как есть (`StateView` loading / empty / error по `loading`, `servers.length`, `err`).
- Импорт: ошибка отображается через `StateView`/чип `bad` (см. 4.2), загрузка — чип `warn` `Importing…`.
- Проверка: «загрузка» = чип `warn` `Checking…` на конкретном сервере.

## 6. Критерии приёмки

1. **Импорт:** при клике видно «идёт импорт», по завершении — стойкий итог `Imported N` (или `Nothing new to import` при 0), при сбое — видимая ошибка на экране (не только toast). Кнопка не кликается повторно во время выполнения.
2. **Проверка серверов:** у каждого сервера понятный статус-чип — `Checking…` во время проверки, `Reachable` (зелёный) при успехе, `Unreachable` (красный) при сбое; до проверки статуса нет (или нейтральный `Untested`). Кнопка `Test` заблокирована во время проверки.
3. **Состояния:** пусто/ошибка/загрузка списка работают как раньше; новые куски используют те же `StateView`/чипы.
4. **`npm run check:ds`** проходит (зелёный).
5. **`npm run typecheck` / `tsc -b`** без ошибок.
6. **`npm test`** (vitest) — без новых падений.
7. Все подписи в UI на английском.

## 7. План проверки

- `cd web && npm run check:ds`

- `cd web && npm test`
- Ручная проверка экрана Connectors: импорт (успех/ошибка/0 создано) и Test (доступен/недоступен/во время проверки).

## 8. Риски / открытые вопросы

- **Реальных «шагов» импорта нет** — бэкенд отдаёт только `{ created }`. «Прогресс» = индикация running на фронте; это сознательное упрощение в рамках «только web».
- Если где-то в тестах есть привязка к старому тексту `imported N` или `· ok` — поправить ассерты (отдельного `Connectors.test` не найдено; проверить на этапе impl/qa).
