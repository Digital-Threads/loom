# SDD — Остановка выполняющейся задачи

**Дата:** 2026-06-16
**Задача (журнал):** tj-cjb70hqays
**Класс:** feature (малая, аддитивная)

## 1. Цель

Дать пользователю кнопку «стоп» для задачи, которая прямо сейчас выполняется.
Когда задача в статусе `running`, её фоновый run и живой процесс сессии Claude
работают, и сейчас прервать это нельзя. Добавляем endpoint и кнопку ⏹ Stop.

## 2. Поведение (приёмочные критерии)

### Backend — `POST /api/tasks/:id/stop`
1. Задачи с таким `id` нет → `404` (`{ error: "not found" }`).
2. Задача есть, у неё есть активный run (`status === "running"`) → останавливаем
   этот run через `RunManager.stop(runId)` **и** убиваем живой процесс сессии
   через `sessionLauncher.stop(sessionId)`. Ответ `{ ok: true }`.
3. Задача есть, активного run нет → ничего не останавливаем, всё равно
   `{ ok: true }`. Идемпотентно: повторный «стоп» не падает.

### Frontend — кнопка ⏹ Stop
4. В шапке задачи (`TaskView`) кнопка ⏹ Stop видна, когда у задачи статус
   `running`. Клик → `client.stopTask(id)`, затем обновление детали задачи.
5. На карточке доски (`Board`) кнопка ⏹ видна, когда `card.status === "running"`.
   Клик → `client.stopTask(id)`, затем обновление доски. Клик по кнопке не
   открывает задачу (`stopPropagation`, как у кнопки удаления).

### Тесты
6. Endpoint останавливает run: мок `RunManager` с `list()` (возвращает running-run
   нужной задачи) и spy-методом `stop` → после POST `stop` вызван с этим `runId`,
   `sessionLauncher.stop` вызван, ответ `200 { ok: true }`.
7. `404` на неизвестной задаче.
8. (Доп.) Идемпотентность: задача без активного run → `{ ok: true }`, `stop` не вызван.

## 4. Дизайн

### 4.1 `RunManager.stop`

В интерфейсе (рядом с `sendInput`):
```ts
/** Останавливает run: помечает запись терминальной и снимает input-handler.
 *  Возвращает false, если run неизвестен или уже не выполняется. Фактическую
 *  работу обрывает убийство живого процесса сессии (sessionLauncher.stop) — после
 *  него задача run-а сама осёдет; этот метод сразу делает статус честным для
 *  доски/SSE/active-run. */
stop(runId: string): boolean;
```

В `createRunManager` (рядом с `sendInput`):
```ts
stop: (runId) => {
  const rec = runs.get(runId);
  if (!rec || rec.status !== "running") return false;
  rec.status = "failed";
  rec.error = "stopped by user";
  inputHandlers.delete(runId);
  return true;
},
```

**Обоснование.** Промис задачи run-а нельзя отменить напрямую (нет
AbortController) — это сознательное ограничение MVP, как и in-memory state в
комментарии к файлу. Реальную остановку даёт `sessionLauncher.stop(sessionId)`:
живой процесс умирает → `sessionSend` бросает → задача run-а уходит в `catch` →
`status = "failed"`. `RunManager.stop` нужен, чтобы (а) статус run-а стал
терминальным **сразу** (тогда `GET active-run` перестаёт его возвращать, SSE
закрывается), (б) endpoint имел явную точку, которую мокает тест.

### 4.2 Endpoint

Ставим рядом с `DELETE /api/tasks/:id` (api.ts L628) — тот же приём 404 и
duck-typed `sessionLauncher.stop`:
```ts
// Stop the task's active run + its live session process. Idempotent: no active
// run → still { ok:true }. 404 only when the task itself is unknown.
app.post("/api/tasks/:id/stop", (c) => {
  const id = c.req.param("id");
  if (!getTask(db, id)) return c.json({ error: "not found" }, 404);
  const run = rm.list().find((r) => r.taskId === id && r.status === "running");
  if (run) rm.stop(run.runId);
  const sid = getTaskSession(db, id).sessionId;
  (sessionLauncher as { stop?: (s: string) => void }).stop?.(sid ?? "");
  return c.json({ ok: true });
});
```
`rm` и `sessionLauncher` уже связаны в `createApi` (L234, L447). Поиск running-run —
тот же, что в `GET active-run` (L1245).

## 5. Риски и ограничения

- **Run не отменяется по-настоящему** до смерти процесса сессии — это MVP-предел
  (см. 4.1). Если живого процесса нет (или мок-launcher без `stop`), `rm.stop`
  всё равно делает статус терминальным.
- **Durable-строка run в БД** (`completeRun`) обновляется штатно — при оседании
  задачи run-а в run-manager (persist.settle) или при загрузке через
  `reconcileInterruptedRuns`. `rm.stop` саму строку не трогает. Для текущей задачи
  это приемлемо (доска показывает статус задачи, не run-а); если потребуется
  немедленно чинить и БД — отдельная доработка.
- **Edge:** если после `stop` задача run-а всё же завершится `done`, статус
  перетрётся на `done`. На практике убитый процесс даёт `failed`, поэтому
  расхождение маловероятно.
- `sessionId` может быть пустым (сессия ещё не создана) — `stop?.(sid ?? "")`
  безопасно, как в delete/switch-profile.
