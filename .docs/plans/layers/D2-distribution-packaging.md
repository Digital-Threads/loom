# SDD — D2.1 + D2.3: Packaging & Global Install of Loom (first release)

Status: spec
Owner: distribution
Scope class: chore (доводка дистрибуции, без новых фич)
Related code: `package.json`, `src/cli.tsx`, `src/cli/serve-args.ts`, `src/web/server.ts`,
`scripts/copy-manifests.mjs`, `web/vite.config.ts`, `web/package.json`

---

## 1. Цель (одним предложением)

Сделать так, чтобы `bun i -g @digital-threads/loom` (и `npm i -g …`) ставил
постороннему пользователю **полный рабочий Loom** — API + веб‑интерфейс + все слои —
а не обрубок, и при этом локальная разработка (на `file:../`) продолжала работать
без изменений.

Это не новая функциональность. Мы доводим распространение уже существующего кода.

---

## 2. Проблема

`package.json` тянет пять соседних модулей по относительным путям:

```
@digital-threads/aimux           file:../aimux
@digital-threads/loom-knowledge  file:../knowledge
@digital-threads/loom-swarm      file:../swarm
@digital-threads/loom-quality    file:../quality
@digital-threads/loom-security   file:../security
```

Локально это работает: модули подключены симлинками в `node_modules`
(`@digital-threads/aimux → /home/shahinyanm/www/loom/aimux`). Но `file:../` — это путь
на машине разработчика. У чужого пользователя такой папки нет → при глобальной
установке эти зависимости не разрешаются и половина Loom отсутствует.

Дополнительно есть две скрытые поломки упаковки (см. §6):
- собранный фронтенд (`web/dist`) **не попадает в пакет** — в `files` его нет;
- Loom фактически требует среду **Bun**, и это нигде не заявлено.

---

## 3. Что уже готово (проверено чтением кода)

Чтобы не делать лишнего:

- **bin и флаги (scope #3) — по факту готовы.** `bin: { loom: "dist/cli.js" }` уже есть;
  `src/cli/serve-args.ts:8` парсит `--port`, `--no-open`, `--project`; `runServe` в
  `src/cli.tsx` их применяет (меняет рабочую папку на `--project`, открывает браузер,
  если не `--no-open`). Остаётся проверить тестом и описать в README.
- **build‑пайплайн (scope #2) — в основном собран.** Скрипты:
  `build:host` (`tsc -p tsconfig.json`), `build:web` (`vite build web`),
  `copy:manifests` (копирует `plugin.json` builtin‑плагинов в `dist`), объединяющий
  `build`, и `prepublishOnly: "bun run build"`.
- Все пять sibling‑модулей **уже оформлены как публикуемые пакеты**: у каждого есть
  `exports`, `files: ["dist"]`, версия, собственный `build` (tsc) и `prepublishOnly`.
  То есть к публикации их форма готова.

---

## 4. Решение: стратегия развязки зависимостей

### Выбор: публикация в реестр + semver‑версии в зависимостях

В `dependencies` пять `file:../` заменяются на **зафиксированные semver‑версии**
уже опубликованных пакетов (`@digital-threads/aimux: ^0.13.0`, остальные `^0.1.0`).
Конечный пользователь скачивает их из реестра вместе со всеми транзитивными
зависимостями — npm/bun делают это сами, включая нативный `better-sqlite3`.

### Как при этом не сломать локальную разработку

Жёсткое требование: «локальная разработка сейчас на `file:../` — её не ломать».
Поэтому **закоммиченный `package.json` остаётся с `file:../`** (байт‑в‑байт как сейчас),
а на публикацию манифест **временно подменяется** скриптом жизненного цикла
(`prepack` → подменить пять `file:../` на pinned semver; `postpack` → вернуть обратно).
`bun install` при разработке `prepack` не запускает, значит dev‑окружение не меняется
вообще. Подмену выполняет скрипт, реальную публикацию запускает человек.

### Почему не другие варианты

- **Бандлить модули в `dist`** (esbuild/tsup) — отклонено: `better-sqlite3` это нативный
  аддон, его нельзя чисто вшить в один JS; `aimux` тянет `commander`, `ink`, `ink-ui`,
  `react`, `yaml` и их транзитивное дерево — бандлинг всего этого хрупок и дублирует
  зависимости, тогда как менеджер пакетов решает это бесплатно.
- **Только workspaces как механизм поставки** — отклонено: workspaces решают линковку
  при разработке, а не то, что пользователь скачивает из реестра; к тому же корень
  workspace лёг бы вне репозитория loom‑host (siblings — отдельные репозитории).

> Предусловие, которое выполняет человек: пять `@digital-threads/loom-*` должны быть
> реально опубликованы в реестр до того, как опубликованный loom‑host станет
> устанавливаемым. Мы готовим конфиг, скрипты и порядок — публикуем не мы.

## 5. Среда выполнения (важное ограничение)

Loom требует **Bun** как среду исполнения:
`src/web/server.ts` использует `Bun.serve` и `hono/bun`, shebang `src/cli.tsx` —
`#!/usr/bin/env bun`. Значит Bun обязателен **и для `npm i -g`**, не только для `bun i -g`.
README должен явно указать Bun (минимальную версию) как предусловие, иначе у
пользователя без Bun запуск `loom` упадёт на shebang.

## 6. Объём работ (по пунктам)

### 6.1 Зависимости и публикация
- Добавить release‑prep скрипт (`scripts/prepare-publish.mjs` или пара `prepack`/`postpack`),
  который подменяет пять `file:../` на pinned semver в манифесте только на время
  pack/publish и возвращает исходный вид после.
- Зафиксировать соответствие имя↔версия для пяти пакетов в одном месте (в скрипте).
- `prepublishOnly: "bun run build"` сохранить.

### 6.2 Упаковка фронтенда
- Добавить `"web/dist"` в `files` (сейчас `["dist", "README.md"]`). Путь
  `server.ts → webDistDir() = <root>/web/dist` уже совпадает с тем, куда кладёт сборку
  `vite build web` → менять `server.ts` не нужно.
- Релизная сборка должна прогонять веб‑гейт `check:ds` (живёт в `web/package.json`),
  чтобы артефакт собирался только на зелёном дизайн‑системном чеке.

### 6.3 bin `loom`
- Подтвердить тестом, что `--port`, `--no-open`, `--project` работают (флаги уже есть).
  Если CLI‑теста на парсинг нет — добавить его (TDD: тест на `parseServeArgs`).

### 6.4 README
- Раздел «Установка и запуск»: предусловие Bun; команды
  `bun i -g @digital-threads/loom` и `npm i -g @digital-threads/loom`; запуск
  `loom` / `loom serve --port N --no-open --project <path>`; что открывается в браузере.

## 8. Критерии приёмки

1. `npm pack` (dry‑run) даёт tarball, в котором: `dist/` (включая `cli.js` и
   скопированные `plugin.json`), `web/dist/`, `README.md`; а пять зависимостей в
   упакованном `package.json` — pinned semver, **без** `file:../`.
2. После `npm pack` закоммиченный `package.json` снова содержит `file:../`
   (подмена обратима, dev‑окружение не изменилось).
3. Локально `bun install` + `bun run dev` и существующие тесты (`vitest run`,
   `smoke:bun`) проходят как прежде.
4. `loom serve --port <N> --no-open --project <dir>` поднимает сервер на нужном порту,
   браузер не открывает, работает в каталоге проекта — подтверждено тестом.
5. Веб‑гейт `check:ds` зелёный; релизная сборка падает, если он красный.
6. README содержит работающие инструкции с явным предусловием Bun.

## 9. Открытые вопросы

- **Версии siblings**: фиксируем `^` или точные версии? Точные надёжнее для
  воспроизводимости первого релиза; уточнить на стадии rd/impl.
- **Порядок публикации**: человек публикует пять `@digital-threads/loom-*` до loom‑host —
  зафиксировать как чек‑лист в README/release‑заметке.
- **prepack vs отдельный скрипт**: `prepack`/`postpack` автоматичнее, но `postpack`
  не вызывается при `npm publish` в части версий npm — на стадии rd проверить
  поведение конкретной версии и при необходимости использовать явный
  `prepare-publish` + восстановление.
